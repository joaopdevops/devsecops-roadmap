# Lab 2.1 — Endereçamento IPv4

**Parte 1:** 2026-05-01
**Parte 2:** 2026-05-02
**Fase:** Redes Fundamentos

---

## Objetivo

Entender a estrutura do IPv4, converter binário↔decimal, classificar endereços e calcular subnets na mão. É aqui que o Mês 01 fica de verdade difícil.

---

## Parte 1 — Estrutura e classificação (01/05)

### Estrutura do IPv4

- 32 bits = 4 octetos de 8 bits
- Cada octeto: 0 a 255 (porque 2⁸ = 256, máximo `11111111` = 255)
- Tabela mestra dos pesos de bit: **128 – 64 – 32 – 16 – 8 – 4 – 2 – 1**

### Conversão binário ↔ decimal

Método: somar os pesos das posições com bit 1.

Exemplo — `01001011`:
```
0×128 + 1×64 + 0×32 + 0×16 + 1×8 + 0×4 + 1×2 + 1×1 = 75
```

Exemplo reverso — 65 → `01000001`:
```
65 = 64 + 1 → bits nas posições 64 e 1 = 01000001
```

### Classes de endereço

| Classe | 1º octeto | Bits iniciais | Uso |
|---|---|---|---|
| A | 1–126 | 0... | redes gigantes |
| B | 128–191 | 10... | redes médias |
| C | 192–223 | 110... | redes pequenas |
| D | 224–239 | 1110 | multicast |
| E | 240–255 | 1111 | reservado/experimental |

**Loopback:** 127.0.0.1 (testa a pilha de rede do próprio dispositivo — não sai pela placa)

### RFC 1918 — Endereços privados

| Classe | Faixa privada |
|---|---|
| A | 10.0.0.0 – 10.255.255.255 |
| B | **172.16.0.0 – 172.31.255.255** (atenção: não é todo 172, só 16 a 31) |
| C | 192.168.0.0 – 192.168.255.255 |

IPs privados não roteiam na internet — precisam de NAT para sair.

### Endereços especiais

| Endereço | Significado |
|---|---|
| 0.0.0.0 | rota default (qualquer origem / qualquer destino) |
| 127.0.0.1 | loopback |
| 169.254.x.x | APIPA — DHCP falhou, SO atribuiu automaticamente |
| 255.255.255.255 | broadcast limitado (só na rede local) |
| x.x.x.255 em /24 | broadcast da rede (ex: 192.168.1.255) |

## Parte 2 — Subnetting matemático (02/05)

### O mecanismo

`/24` = 24 bits de rede, **8 bits de host**. Hosts úteis: **2⁸ − 2 = 254**

Fórmula universal: **2^(bits de host) − 2**

O "−2" desconta:
- Endereço de rede (todos bits de host = 0): ex. `192.168.1.0`
- Broadcast (todos bits de host = 1): ex. `192.168.1.255`

### Tabela de referência

| Máscara | Bits de host | Hosts úteis |
|---|---|---|
| /24 | 8 | 254 |
| /25 | 7 | 126 |
| /26 | 6 | 62 |
| /27 | 5 | 30 |
| /28 | 4 | 14 |
| /30 | 2 | 2 |

**/30 = enlace ponto a ponto.** Dois roteadores conectados: cada lado precisa de 1 IP. 2 hosts é exatamente o suficiente — usar /24 seria desperdiçar 252 endereços.

### Subnet sizing — princípio

Escolher a **menor máscara que comporta o requisito** sem desperdiçar bloco.

Exemplo Equinix (3 redes):
- 200 servidores → /24 (254 hosts)
- 50 servidores → /26 (62 hosts — /27 teria só 30, insuficiente)
- 10 servidores → /28 (14 hosts — /29 teria só 6, insuficiente)

## Erros cometidos

| Erro | Causa | Correção |
|---|---|---|
| `172.20.5.10` → acertou Classe B mas não disse privado | Não verificou se estava na faixa 172.16–31 | Sempre checar faixa privada do 172, não só "começa com 172" |
| Justificou 254 hosts em /24 com "é Classe C" | Associação decorada sem mecanismo | O cálculo é sempre 2^(bits de host) − 2 — independente de classe |

---

## Ponte DevOps

- **VPC na AWS:** você define subnets com exatamente essa lógica — /24 pra rede de aplicação, /28 pra sub-rede de banco isolado, /30 pra enlace entre VPCs.
- **Security groups + subnet isolation:** separar ambientes por subnet (/26 pra prod, /26 pra staging) é prática padrão de SRE.
- **Terraform:** `cidrsubnet("10.0.0.0/16", 8, 1)` — calcula subnet programaticamente, mas você precisa entender o /notation pra não dimensionar errado.
